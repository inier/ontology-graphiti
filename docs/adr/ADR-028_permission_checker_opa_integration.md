# ADR-028: 采用OPA作为统一权限校验引擎

## 状态

**已接受** | **创建日期**: 2026-04-13 | **最后更新**: 2026-04-13

## 上下文

Graphiti作为领域仿真AI Agent平台，需要实现细粒度的访问控制：
1. **工具调用权限**: 控制哪些用户可以执行哪些工具
2. **数据访问权限**: 控制用户对领域数据、仿真结果、配置信息的访问
3. **系统操作权限**: 控制Agent管理、系统配置、监控查看等操作
4. **项目隔离权限**: 不同项目之间的资源访问隔离
5. **时间/IP限制**: 基于时间和IP地址的访问控制

当前系统面临以下挑战：
- **权限逻辑分散**: 权限检查代码分散在各个业务模块
- **策略管理复杂**: 权限策略难以统一管理和更新
- **缺乏审计追踪**: 权限决策过程缺乏完整的审计记录
- **扩展性差**: 新增权限类型需要修改大量代码
- **测试困难**: 权限逻辑与业务逻辑耦合，测试复杂

现有解决方案包括：
1. **硬编码权限检查**: 在代码中直接编写权限逻辑
2. **基于角色的访问控制（RBAC）**: 使用角色进行权限管理
3. **基于属性的访问控制（ABAC）**: 使用属性进行细粒度控制
4. **策略即代码（Policy as Code）**: 使用声明式策略语言

## 决策

**采用Open Policy Agent (OPA)作为Graphiti系统的统一权限校验引擎，实现声明式、细粒度、可审计的访问控制。**

### 核心决策点：
1. **策略与代码分离**: 权限策略使用Rego语言编写，与业务代码解耦
2. **统一策略引擎**: 所有权限检查通过统一的OPA引擎执行
3. **细粒度控制**: 支持用户、角色、资源、环境、时间等多维度属性
4. **实时策略更新**: 策略变更实时生效，无需重启系统
5. **完整审计追踪**: 记录所有权限决策的完整上下文

## 替代方案分析

### 方案A: 硬编码权限检查（当前状态）
- **优点**: 
  - 简单直接，易于理解
  - 编译时检查，类型安全
  - 性能开销小
- **缺点**:
  - 权限逻辑分散，难以维护
  - 策略变更需要代码修改和部署
  - 缺乏统一的审计和监控
  - 扩展性差，新增权限类型复杂

### 方案B: 基于角色的访问控制（RBAC）
- **优点**:
  - 成熟的标准，广泛采用
  - 角色管理直观，易于理解
  - 权限分配简单
- **缺点**:
  - 粒度较粗，难以实现细粒度控制
  - 角色爆炸问题（角色数量过多）
  - 难以处理复杂的环境属性
  - 策略表达能力有限

### 方案C: 基于属性的访问控制（ABAC）
- **优点**:
  - 细粒度控制，支持复杂条件
  - 策略表达能力强
  - 适合复杂业务场景
- **缺点**:
  - 实现复杂，需要完整的属性管理系统
  - 性能开销较大
  - 策略管理复杂
  - 缺乏标准化实现

### 方案D: OPA（最终选择）
- **优点**:
  - **策略即代码**: 使用Rego语言，策略可版本控制
  - **统一引擎**: 所有权限检查通过统一引擎执行
  - **细粒度控制**: 支持多维度属性，策略表达能力强
  - **实时生效**: 策略变更无需重启系统
  - **完整审计**: 内置决策日志和追踪
  - **生态完善**: 丰富的工具链和社区支持
- **缺点**:
  - 学习Rego语言有一定成本
  - 需要部署和维护OPA服务
  - 性能开销比硬编码方案大

### 方案E: Casbin
- **优点**:
  - 轻量级，易于集成
  - 支持多种访问控制模型
  - 多语言支持良好
- **缺点**:
  - 策略表达能力相对有限
  - 审计和监控功能较弱
  - 社区生态相对较小

## 影响

### 正面影响
1. **提高可维护性**: 权限策略集中管理，与业务代码解耦
2. **增强灵活性**: 策略变更实时生效，无需代码修改
3. **改善安全性**: 细粒度控制和完整审计追踪
4. **提升开发效率**: 统一的权限检查接口
5. **加强合规性**: 满足安全合规要求的审计能力

### 负面影响
1. **系统复杂性增加**: 需要部署和维护OPA服务
2. **性能开销**: 额外的网络调用和策略评估开销
3. **学习成本**: 开发人员需要学习Rego语言
4. **调试难度**: 策略调试需要专门工具

### 权限检查流程
1. **请求接收**: 接收权限检查请求
2. **输入构建**: 构建OPA输入数据（用户、资源、操作、环境）
3. **策略查询**: 调用OPA引擎执行策略查询
4. **决策解析**: 解析OPA返回的决策结果
5. **审计记录**: 记录权限决策的完整上下文
6. **结果返回**: 返回允许/拒绝决策

### 核心策略示例
```rego
# base.rego - 基础策略包
package graphiti.base

import input.user
import input.resource
import input.action
import input.environment

# 默认拒绝所有请求
default allow = false

# 系统管理员允许所有操作
allow {
    user.roles[_] == "system_admin"
}

# 检查用户是否有指定操作权限
allow {
    # 获取用户角色
    user_role := roles.get_user_role(user.id)
    
    # 获取资源权限要求
    resource_perm_required := get_resource_permission_required(resource.type, resource.id)
    
    # 检查角色权限
    roles.role_has_permission(user_role, resource_perm_required, action)
    
    # 检查环境限制
    environment_check_passed
}

# 环境检查
environment_check_passed {
    # 检查时间限制
    environment.current_time >= environment.start_time
    environment.current_time <= environment.end_time
    
    # 检查IP限制
    environment.client_ip in environment.allowed_ips
}
```

## 示例用例

### 工具调用权限检查
```python
async def check_tool_permission(user_id: str, 
                               tool_name: str,
                               action: str = "execute",
                               context: Optional[Dict] = None) -> bool:
    """检查工具调用权限"""
    
    # 构建OPA输入
    input_data = {
        "user": {
            "id": user_id,
            "roles": await get_user_roles(user_id),
            "attributes": await get_user_attributes(user_id)
        },
        "resource": {
            "type": "tool",
            "id": tool_name,
            "attributes": await get_tool_attributes(tool_name)
        },
        "action": action,
        "environment": context or {}
    }
    
    # 调用OPA检查权限
    result = await opa_client.check_permission(
        policy_name="graphiti/tools_policy",
        input_data=input_data
    )
    
    return result.get("allow", False)
```

### 数据访问权限检查
```python
async def check_data_permission(user_id: str,
                               data_type: str,
                               data_id: str,
                               operation: str,
                               context: Optional[Dict] = None) -> bool:
    """检查数据访问权限"""
    
    # 构建OPA输入
    input_data = {
        "user": {
            "id": user_id,
            "roles": await get_user_roles(user_id),
            "attributes": await get_user_attributes(user_id)
        },
        "resource": {
            "type": data_type,
            "id": data_id,
            "attributes": await get_data_attributes(data_type, data_id)
        },
        "action": operation,
        "environment": context or {}
    }
    
    # 调用OPA检查权限
    result = await opa_client.check_permission(
        policy_name=f"graphiti/{data_type}_policy",
        input_data=input_data
    )
    
    return result.get("allow", False)
```

## 备注

本决策将作为Graphiti系统权限管理的标准架构，所有新的权限检查都应通过OPA引擎执行。现有权限逻辑将按优先级迁移到OPA策略。

**风险提示**: OPA策略的复杂性可能导致策略调试困难，需要配套完善的测试和调试工具。策略性能需要持续监控和优化。
