# ADR-046: Phase 4 采用模块化单体部署架构

## 状态
已接受

## 上下文

ODAP 当前代码组织为单体 Python 包（`odap/`），按分层模块化设计：

```
odap/
├── infra/       # L1 基础设施（Graphiti、OPA、Embedder）
├── biz/         # L2-L3 业务逻辑（Swarm、Ontology、Workspace、Hook...）
├── tools/       # L4 领域工具（Skills）
└── web/         # L5-L6 接口层（API Gateway、WebSocket）
```

架构文档中存在两种部署描述的矛盾：

1. **ARCHITECTURE.md §1.5** 明确约束"小团队（3-5人），无需微服务拆分"
2. **api_gateway/DESIGN.md §8** 描述了"Phase 2+ 微服务"的演进路线
3. **ARCHITECTURE_PLAN.md** 按微服务风格设计了模块间通信

ANOMALY_REPORT I-21 将此列为 🔴 亟需确认项。

### 约束条件

| 维度 | 现实 |
|------|------|
| 团队规模 | 3-5 人，无专职 SRE/DevOps |
| 部署目标 | Docker Compose 生产环境 |
| 当前代码 | 单体包，所有模块 in-process |
| 模块边界 | 已通过 Python 包 + 接口抽象约束 |
| 运维能力 | 无 K8s 经验，无 Service Mesh 基础设施 |

## 决策

**Phase 4 采用模块化单体（Modular Monolith）部署架构**：

1. **单一进程**：所有业务模块运行在同一 FastAPI 进程中
2. **模块边界通过代码约束**：接口抽象（IGraphitiClient、IOPAManager 等）+ Python 包结构
3. **外部服务独立部署**：Neo4j、OPA 作为独立 Docker 容器
4. **Docker Compose 编排**：主进程 + Neo4j + OPA 三个核心容器
5. **预留微服务演进能力**：保持纯 Python 接口抽象，未来可按模块拆分为独立进程

### 部署拓扑

```
Docker Compose
├── odap-api        # FastAPI 主进程（所有模块 in-process）
├── neo4j           # 图数据库
├── opa             # 策略引擎
└── (optional)
    ├── redis       # 缓存（按需引入）
    └── nginx       # 反向代理 + 静态资源
```

### 模块边界约束规则

1. **依赖方向**：`web/` → `biz/` → `infra/`，禁止反向依赖
2. **接口隔离**：跨模块调用必须通过接口（`I*` 前缀抽象类），不直接引用实现类
3. **事件解耦**：模块间异步通信通过 EventBus（Hook 系统），不直接调用
4. **配置隔离**：每个模块有独立配置 schema，不共享配置对象

## 后果

### 变得更容易

- **开发效率**：单仓库、单进程调试、无网络序列化开销
- **部署简单**：Docker Compose 三容器即可，无需 K8s/Swarm
- **事务一致性**：in-process 调用天然支持事务，无需分布式事务
- **调试友好**：全链路在一个进程内，日志/追踪无需跨服务关联
- **可逆性高**：模块边界清晰，未来可按模块拆出独立进程

### 变得更难

- **独立扩缩容**：无法按模块独立扩缩，只能整体水平扩展
- **故障隔离**：单进程内模块故障可能传染（OOM、死循环）
- **技术异构**：所有模块必须用 Python，无法按模块选型
- **团队边界**：多人修改同一代码库需协调

### 风险与缓解

| 风险 | 概率 | 缓解 |
|------|------|------|
| 单点 OOM 导致全服务不可用 | 中 | 进程监控 + 自动重启 + 内存限制告警 |
| 模块间耦合逐渐松弛 | 高 | 代码审查强制接口约束 + Arch Unit 测试 |
| 未来需要独立扩缩某模块 | 低 | 按模块拆出独立 FastAPI 服务，接口不变 |

### 微服务演进路径（Phase 5+ 触发条件）

仅在以下条件**同时满足**时考虑拆分：

1. 团队扩大到 8+ 人，需要独立部署节奏
2. 某模块有明确的独立扩缩容需求（如推演引擎 CPU 密集）
3. 有专职 SRE 负责 K8s 运维

拆分策略：将模块从 in-process 调用改为 HTTP/gRPC 调用，接口签名不变。

## 可逆性

**高**。模块化单体 → 微服务 是可逆方向（合回来很难，但拆出去容易）。当前决策保持了所有模块的接口抽象，拆分成本可控。
