# ADR-033: 项目目录结构重构

## 状态
提议

## 日期
2026-04-15

## 上下文

当前项目（Phase 0-3）已完成全部功能开发，52 个 Python 文件实现了图谱管理、Agent 协同、模拟推演、Web 可视化等完整功能。但随着代码量增长，`core/` 目录积聚了 21 个文件，横跨基础设施、业务逻辑、适配层、Web 服务等多个架构层次，导致以下问题：

1. **职责不清晰**：`core/graph_manager.py`（图谱基础设施）和 `core/simulation_engine.py`（业务逻辑）同级放置
2. **模块不内聚**：本体管理相关代码分散在 `ontology/`（3个文件）+ `core/ontology_*`（3个文件）
3. **模拟器碎片化**：模拟推演分散在 `core/simulation_engine.py` + `core/data_ingestion.py` + `core/simulator_web_service.py` + `simulator_ui/`
4. **与 OpenHarness 不兼容**：OpenHarness 使用 `src/openharness/` 模块化结构，当前平铺结构不利于后续集成
5. **根目录散乱**：6 个 HTML + 2 个 PNG 散落在根目录

架构文档 `ARCHITECTURE.md` 定义了四层组件（OpenHarness → Graphiti → Skills → OPA），但代码结构没有与之对齐。

## 决策

采用 **分层模块化目录结构**，将项目代码从 `core/` "大杂烩" 重构为 `odap/` 包，按四层架构组织：

| 层次 | 目录 | 职责 |
|------|------|------|
| L1 适配层 | `odap/adapters/` | OpenHarness、MCP 等外部系统集成 |
| L2 基础设施层 | `odap/infra/` | 图谱、LLM、OPA、Hook、韧性（无业务逻辑） |
| L3 领域工具层 | `odap/tools/` | Skills 注册、9 大工具类别 |
| L4 业务领域层 | `odap/biz/` | 本体管理、工作空间、Agent 协同、模拟推演、权限 |
| Web 层 | `odap/web/` | FastAPI 路由 + 前端 SPA |
| 存储层 | `odap/storage/` | 运行时数据（版本、场景、状态、导出） |

核心业务模块对外暴露清晰 API：

| 模块 | 对外 API | 核心方法 |
|------|---------|---------|
| 本体管理 | `OntologyManager` | create_workspace, update, reset, switch_version, hot_write, query, export, import |
| 工作空间 | `WorkspaceManager` | create, delete, list, switch, get_config |
| Agent 协同 | `BattlefieldSwarm` | initialize, execute_mission, shutdown |
| 模拟推演 | `SimulatorEngine` | create_scenario, run_simulation, pause, resume |
| Web 服务 | `FastAPI Router` | RESTful API + WebSocket |

详细方案见 `docs/RESTRUCTURE_PLAN.md`。

## 备选方案

### 方案 A：保持现有结构，仅清理根目录
- **优势**：零迁移成本
- **劣势**：不解决核心问题（core/ 大杂烩、模块不内聚），技术债持续累积

### 方案 B：完全按 OpenHarness 目录复制
- **优势**：与 OpenHarness 100% 兼容
- **劣势**：OpenHarness 是通用 Agent 框架，其结构不完全适合 ODAP 的本体驱动分析场景（缺少 ontology/、workspace/ 等业务模块）

### 方案 C（采纳）：分层模块化 + OpenHarness 兼容
- **优势**：
  - 清晰的层次划分和依赖方向
  - 业务模块内聚，对外 API 清晰
  - 与 OpenHarness 结构对齐（`src/` + 子包），便于后续集成
  - 渐进式迁移，向后兼容 shim 保障平滑过渡
- **劣势**：
  - 需要约 10-16 天迁移工作量
  - 需要更新所有 import 路径

## 后果

### 正面
- **可维护性大幅提升**：每个模块职责单一，修改影响范围可控
- **可测试性提升**：基础设施层可独立测试，Mock 业务层更简单
- **OpenHarness 兼容**：结构对齐为后续 Phase 4 集成打好基础
- **新功能开发加速**：新增模块（如 workspace）有清晰的归属位置
- **依赖方向可控**：禁止反向依赖，避免循环引用

### 负面
- **迁移成本**：约 10-16 天（6 个 Phase）
- **import 路径变更**：需要 shim 过渡期（建议保留 2 个版本）
- **文档更新**：ARCHITECTURE.md、测试文件、README 均需同步更新

### 风险缓解
- 使用 `git mv` 保持文件历史
- 每个迁移 Phase 后运行全量测试
- 向后兼容 shim（re-export + DeprecationWarning）
- 使用 `import-linter` 验证依赖方向
